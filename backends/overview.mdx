## What is a backend?

A **backend** is the feature extraction engine that capa uses to analyze a binary. When capa runs, it loads the target file with a backend, which disassembles instructions, parses file structures, and traces execution behavior. The backend emits **features** ã observations about the binary ã that capa's rule engine then matches against the rule set.

Different backends provide different capabilities:

- **Static backends** (Vivisect, IDA Pro, Ghidra, Binary Ninja, BinExport2, pefile, .NET/dnfile) disassemble the binary and extract code-level features without running it.
- **Dynamic backends** (CAPE, DRAKVUF, VMRay) extract features from a sandbox report generated by actually executing the sample.
- **Utility backends** (Freeze) replay a cached feature set so you can re-run rules without re-analyzing.

## Supported backends

| Backend | Flag value | Use case | Supported file types |
|---------|------------|----------|----------------------|
| Vivisect | `vivisect` | Default static analysis | PE, ELF, SC32, SC64 |
| IDA Pro | `ida` | Interactive analysis via IDA Pro or idalib | Any IDA-supported format |
| Ghidra | `ghidra` | Open-source disassembly via PyGhidra | Any Ghidra-supported format |
| Binary Ninja | `binja` | Analysis via Binary Ninja API | Any Binary Ninja-supported format, `.bndb` databases |
| BinExport2 | `binexport2` | Cross-tool analysis via BinExport2 format | `.BinExport` files |
| pefile | `pefile` | Fast PE metadata extraction only | PE files |
| .NET (dnfile) | `dotnet` | Managed .NET code analysis | .NET PE assemblies |
| CAPE | `cape` | Dynamic sandbox report analysis | CAPE JSON reports (`.json`, `.json_`, `.json.gz`) |
| DRAKVUF | `drakvuf` | Dynamic sandbox report analysis | DRAKVUF log files (`.log`, `.log.gz`) |
| VMRay | `vmray` | Dynamic sandbox report analysis | VMRay analysis archives (`.zip`) |
| Freeze | `freeze` | Replay previously extracted features | `.frz` files |

## Selecting a backend

Pass the `-b` / `--backend` flag with the backend name to select a specific backend:

```bash
capa --backend vivisect sample.exe
capa --backend ida sample.idb
capa --backend ghidra sample.exe
capa --backend binja sample.bndb
capa --backend cape report.json
```

## Automatic backend detection

When you do not pass `--backend`, capa inspects the input file and selects an appropriate backend automatically:

- CAPE JSON reports ã `cape` backend
- DRAKVUF `.log` files ã `drakvuf` backend
- VMRay archives ã `vmray` backend
- `.NET` PE files ã `dotnet` backend
- Freeze files ã `freeze` backend
- BinExport2 files ã `binexport2` backend
- Everything else ã `vivisect` backend (default static analysis)

<Note>
Auto-detection is based on file content and magic bytes, not file extension. You can override it at any time with `--backend`.
</Note>

## Choosing the right backend

<CardGroup cols={2}>
  <Card title="Static analysis" icon="magnifying-glass" href="/backends/overview">
    Use **Vivisect** for most PE and ELF files out of the box. Switch to **IDA Pro**, **Ghidra**, or **Binary Ninja** to leverage the disassembly you already have open in those tools.
  </Card>
  <Card title="Packed or obfuscated samples" icon="lock" href="/backends/dynamic-analysis">
    Static analysis struggles with packed samples. Run the sample in a sandbox and use the **CAPE**, **DRAKVUF**, or **VMRay** backend against the generated report.
  </Card>
  <Card title="Interactive rule building" icon="wrench" href="/backends/ida-plugin">
    The **IDA Pro plugin** lets you navigate directly to matched addresses and build new rules from extracted features, all within IDA's UI.
  </Card>
  <Card title="Open-source disassembly" icon="code" href="/backends/ghidra-plugin">
    The **Ghidra backend and plugin** give you capa's capabilities on top of Ghidra's free, open-source analysis engine.
  </Card>
</CardGroup>
